# PRD：基于业务组件的页面拼装与调用规范

| 文档属性 | 说明 |
|----------|------|
| 状态 | 草案 |
| 范围 | 多项目复用的业务 UI 与业务页面之间的集成方式 |
| 相关决策 | 表格区采用**外壳自研 + 内部按需使用 antd/vc-design 小组件**（与当前 demo 表格实例一致） |

---

## 1. 背景与问题

### 1.1 背景

- 设计在 Figma 中按**功能区块**（导航区、筛选区、切换区、操作区、表格区等）交付页面。
- 期望这些区块在代码侧对应**可发布、可版本管理**的业务组件，在多个业务项目中复用。
- 业务页面除拼装业务组件外，还需按场景插入**基础组件**（表单控件、反馈组件等）。

### 1.2 当前痛点（待本 PRD 指导解决）

- Demo 中的「组件实例」与业务项目中的「可调用组件」边界不清，易出现复制粘贴 Demo 代码、样式与 token 漂移。
- 缺少统一的**页面级拼装约定**（数据从哪进、扩展点在哪、谁负责布局与路由）。
- 多项目复用时，**依赖版本、破坏性变更**缺少明确契约。

---

## 2. 目标与成功标准

### 2.1 产品目标

1. **业务开发者**能在不阅读实现细节的前提下，通过**稳定 API + 文档**完成常见列表页/详情框架页的搭建。
2. **业务组件库**与 **vc-design / Ant Design** 的职责边界清晰，减少页面层随意覆盖样式导致的与设计稿不一致。
3. 支持**多项目、多版本**依赖同一套业务组件，升级路径可预期。

### 2.2 成功指标（建议）

| 指标 | 说明 |
|------|------|
| 集成耗时 | 标准列表页骨架（含筛选+切换+操作+表格壳）从 0 到可联调 UI 的时间可量化下降 |
| API 稳定性 | 业务组件 major 版本内，公开 API 破坏性变更次数与迁移文档完备率 |
| 一致性 | 新页面走拼装规范后，设计走查中与 Figma 区块级偏差率下降 |

---

## 3. 用户与场景

### 3.1 主要用户

| 角色 | 诉求 |
|------|------|
| 业务前端 | 快速拼页、类型安全、少踩样式坑 |
| 前端架构 / 组件维护者 | 可发布、可演进、依赖与 peer 范围清晰 |
| 设计 | Figma 模块与代码组件有可追溯映射 |

### 3.2 核心场景

- **S1**：新建列表页 — 侧栏/顶栏 + 筛选 + Tab 切换 + 表格操作条 + 自研表格壳 + 业务数据注入。
- **S2**：在业务组件提供的**插槽**内放入基础组件（如自定义筛选项、额外按钮、表格单元格渲染）。
- **S3**：仅替换数据与配置（列配置、筛选项配置），不改区块结构。
- **S4**：跨项目升级业务组件 minor/patch，页面改动量可控。

---

## 4. 范围

### 4.1 范围内

- 业务组件库的**对外调用方式**（包形态、入口、命名）。
- **页面拼装分层**与**推荐目录/组合顺序**。
- **数据与回调**的输入输出约定（受控/非受控、配置驱动）。
- **扩展点**（slots / renderProps / children）的命名与使用原则。
- 与 **Figma 模块**的对应关系（文档层要求）。
- **表格区**：保持「外壳自研 + 内部小组件」策略下的 API 形状建议（列/行数据、工具条插槽等）。

### 4.2 范围外（可另立 PRD）

- 具体每一个业务组件的像素级 Figma 还原清单（可由各组件 Spec 承接）。
- 后端接口协议、权限模型、路由方案的具体选型。
- 非 React 技术栈适配。

---

## 5. 设计原则

1. **区块即组件**：Figma 中独立成块的区域，优先对应单一业务组件；避免「巨型页面组件」难以复用。
2. **配置优于拷贝**：列表页差异尽量体现在 **JSON/TS 配置**（筛选项、Tab 项、表格列）而非复制 JSX 结构。
3. **演示与产物分离**：Demo 仅用于说明与调试；**可 import 的导出**不得依赖 Demo 专用控件（如仅用于 playground 的 Slider 配置面板）。
4. **样式单源**：业务组件外观以 **vcTokens**（及少量组件级 CSS）为准；页面层默认不覆盖业务组件内部布局样式。
5. **组合优于继承**：扩展通过 **children / left / right / renderCell** 等显式插槽，避免 boolean 爆炸。
6. **peer 清晰**：`react`、`react-dom`、`antd`、`vc-design` 作为 peerDependencies，版本范围在文档与 changelog 中明示。

## 6. 功能需求

### 6.1 包与入口

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| F-1 | 业务组件以**独立 npm 包**（或 monorepo 子包）发布，包名固定、主入口导出稳定 | P0 |
| F-2 | 提供 **TypeScript 类型**与 **按需/全量**导出策略说明 | P0 |
| F-3 | 每个业务组件在文档中有：**用途、Figma 对应模块、Props 表、示例代码** | P0 |

### 6.2 页面拼装模型

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| F-4 | 定义**推荐页面骨架顺序**（例如：Layout → 筛选 → 切换 → 操作 → 表格壳），与常见 Figma 结构一致 | P0 |
| F-5 | 支持在骨架任意**合法插槽**内插入基础组件或业务子片段，且不破坏区块级间距与对齐规范 | P0 |
| F-6 | 提供可选 **PageShell / ListPageTemplate**（名称可议）：仅负责区块顺序与留白，不包含业务数据 | P1 |

### 6.3 数据与状态

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| F-7 | 筛选、切换、分页等常见交互支持**受控模式**（value + onChange），便于与 URL、全局状态同步 | P0 |
| F-8 | 表格区支持**数据与列定义由页面传入**；排序/筛选/加载态等行为在 PRD 附录或组件 Spec 中逐项定义 | P0 |
| F-9 | 业务组件**不隐式请求接口**；数据获取留在页面或业务 hooks | P0 |

### 6.4 表格区（自研外壳策略）

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| F-10 | 表格**布局与交互外壳**自研，与当前 demo 满意度对齐；单元格内可使用 antd/vc-design **小组件**（如 Checkbox、Tag、Button） | P0 |
| F-11 | 对外暴露 **列定义 API**（字段、标题、宽度策略、自定义渲染、对齐方式等），类型与运行时校验策略在实现阶段补齐 | P0 |
| F-12 | 工具条/批量操作区与表体**解耦**（可与 OperationBar 组合或由表格壳提供顶栏插槽） | P1 |

### 6.5 多项目复用与升级

| ID | 需求描述 | 优先级 |
|----|----------|--------|
| F-13 | **Changelog** 标注破坏性变更与迁移步骤 | P0 |
| F-14 | major 版本可收紧 peer；minor/patch 不得无故收紧 peer 范围 | P0 |
| F-15 | 提供**最低可运行版本矩阵**（React / antd / vc-design） | P1 |

## 7. 非功能需求

| ID | 类别 | 描述 |
|----|------|------|
| NF-1 | 性能 | 表格在约定数据量下的滚动与重绘可接受；大数据量场景需明确是否虚拟列表及由谁提供 |
| NF-2 | 可访问性 | 关键交互（筛选、切换、表格行选择）具备可访问性基线，在组件 Spec 中列出 |
| NF-3 | 可测试性 | 业务组件核心交互可通过稳定 `data-testid` 或文档约定选择器支持 E2E（可选 P1） |
| NF-4 | 安全 | 若存在 `dangerouslySetInnerHTML` 或富文本列，需在组件层注明风险与页面责任边界 |

## 8. 页面调用约定（规范级）

### 8.1 推荐分层

```
业务页面（路由、数据、权限）
  └── 页面布局（Layout，可选来自业务库或应用自有）
        └── 业务区块组件（筛选 / 切换 / 操作 / 表格壳 …）
              └── 基础组件（Input、Select、Button …）—— 优先通过业务组件插槽注入
```

### 8.2 调用方式（示例级要求，非代码实现）

- 页面文件只出现：**导入业务组件、传入配置与回调、组合插槽内容**。
- 与 Figma 不一致的定制：优先提需求给业务组件库（新增变体或插槽），而非在页面写大段覆盖样式。

### 8.3 Figma 映射（文档义务）

- 维护 **模块名 → 组件名 → 关键 Props / 变体** 对照表（可放在组件库文档站或 `docs/`）。
- 设计变更若影响公开 API，走**组件库版本**与**changelog**，并通知消费方。

## 9. 里程碑建议

| 阶段 | 产出 |
|------|------|
| M1 | 确定包名、peer 范围、公开导出清单；1～2 个业务组件从 Demo 抽为可发布 API |
| M2 | 列表页模板 + 筛选/切换/操作/表格壳串联示例项目（最小可运行） |
| M3 | 表格列 API 定稿 + 与自研外壳联调；文档与 Figma 映射表 v1 |
| M4 | 第二个业务项目试点接入 + 收集集成问题，修订本 PRD |

---

## 10. 风险与依赖

- **风险**：业务项目 antd 版本分裂 → 通过 peer 范围与版本矩阵缓解。
- **风险**：表格自研能力滞后于业务（树形、嵌套、复杂合并单元格）→ 在列 API 与路线图中显式标注「支持/不支持/后续版本」。
- **依赖**：vc-design token 稳定；Figma 与 token 对齐流程可持续执行。

---

## 11. 开放问题（需评审拍板）

1. 业务组件包是否与 vc-design **同仓 monorepo** 发布，还是**独立仓库**？
2. 列表页模板是否纳入业务包，还是由各应用自建模板仅引用区块组件？
3. 表格虚拟滚动是否必须在 v1 提供，还是明确 v1 仅支持约定最大行数？

## 12. 附录：术语

| 术语 | 含义 |
|------|------|
| 业务组件 | 带业务语义与布局的复合 UI，多项目复用，依赖 vc-design/antd |
| 基础组件 | Ant Design / vc-design 提供的通用控件 |
| 组件实例（Demo） | 文档与开发环境中用于展示的示例，不等同于生产 API |
| 自研表格外壳 | 非 antd Table 整表封装，自管行列表头布局；内部可选用小组件 |
